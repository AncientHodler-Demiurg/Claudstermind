# Architecture — ChainwebMiningClient

> Big-picture design. File-by-file specifics live in the project's own [`CLAUDE.md`](../../../ChainwebMiningClient/CLAUDE.md); this page captures the shape that takes several reads to see.

## Stack

- **Language:** Haskell (`default-language: Haskell2010`)
- **Build:** Cabal 3.0, `build-type: Custom` (the custom-setup autogenerates `PkgInfo.hs` via `configuration-tools`)
- **GHC:** 8.10.7 / 9.0.2 / 9.2.4 declared; CI also exercises 9.4
- **Key deps:** `configuration-tools` (config + CLI), `http-client` + `http-client-tls` (node API), `streaming-events` (SSE subscription), `cryptonite` (Blake2s), `warp` + `wai` + `deeproute` (on-demand + stratum servers), `sydtest` + `QuickCheck` (tests)
- **Extra:** `cabal.project` pulls `deeproute` from a pinned git commit (not Hackage yet)

## The `Worker` seam

Everything pivots on one type:

```haskell
type Worker = Nonce -> Target -> ChainId -> Work -> IO Work
```

`main/Main.hs` carries a `WorkerConfig` sum type (`CpuWorker | ExternalWorker | StratumWorker | SimulatedMinerWorker | ConstantDelayWorker | OnDemandWorker`) and `withWorker` is a bracket that:

1. **Allocates backend-specific state** — e.g. `StratumWorker` spins up a TCP server, `OnDemandWorker` spins up a Warp server, `CpuWorker` does nothing.
2. **Hands a closure `Logger -> Worker` to the continuation** — the outer loop calls that with a per-thread logger to get a real `Worker`.
3. **Tears the state down on exit.**

The outer loop (`miningLoop`) is worker-agnostic. It does the HTTP / SSE dance and delegates solving to whatever `Worker` it was handed. That's why new modes should add a sum-type constructor + a branch in `withWorker`, never a parallel loop in `main`.

## Execution model

```
                       ┌──────────────────────── chainweb-node ─────────────────────────┐
                       │  GET  /mining/work          (server → client: new work)        │
                       │  POST /mining/solved        (client → server: solution)        │
                       │  GET  /mining/updates       (SSE stream per chain; stale work) │
                       └────────────────────────────────────────────────────────────────┘
                                               ▲              ▲
                                               │              │ one shared UpdateMap
                                               │              │ (SSE subscription per chain)
                                  ┌────────────┴──────────────┴───────────┐
                                  │              miningLoop               │   × _configThreadCount
                                  │   retry(fetch) → worker → retry(post) │
                                  └───────────────────┬───────────────────┘
                                                      │
                                                      ▼
                                        ┌─────────── Worker ───────────┐
             CPU: blake2s in-process  ◄──┤                              ├──►  Stratum: TCP server, JSON-RPC to ASICs
             External: spawn subproc  ◄──┤     one of six backends     ├──►  OnDemand: HTTP POST /make-blocks
             SimulatedMiner: sleep    ◄──┤                              ├──►  ConstantDelay: sleep
                                        └──────────────────────────────┘
```

**Key property:** the `UpdateMap` is shared across all `_configThreadCount` loops. One "work is stale" event cancels every in-flight worker. That's what keeps the threads from wasting cycles on superseded work.

## Config model

`configuration-tools` gives the combined config-file + env + CLI treatment. Convention: field `_configFoo` ↔ JSON key `foo`. Every setting must round-trip through `--print-config` to YAML and back via `--config-file`. Defaults live in `defaultConfig` in `main/Main.hs`.

Config also owns the keygen path: `--generate-key` short-circuits before `run`, emits an Ed25519 pair, and exits.

## Mining modes, by intent

| Mode             | Produces valid blocks? | Intent                                                                    |
| ---------------- | ---------------------- | ------------------------------------------------------------------------- |
| `stratum`        | yes                    | Serve ASIC miners over TCP; competitive mainnet mode                      |
| `external`       | yes                    | Delegate to a GPU subprocess (e.g. `bigolchungus`) per job                |
| `cpu`            | yes                    | Multi-threaded Blake2s_256 in-process; too slow for mainnet but valid     |
| `simulation`     | **no** (PoW disabled)  | Sleep by `--hash-rate` and block difficulty; for difficulty-adjust tests  |
| `constant-delay` | **no**                 | Sleep a fixed `--constant-delay-block-time`; test heartbeat               |
| `on-demand`      | **no**                 | HTTP server on `--on-demand-port`; POST `/make-blocks` to trigger a block |

The non-PoW trio require the node to run with `DISABLE_POW_VALIDATION=1`. The README flags this loudly; so should any new mode in that family.

## Stratum subsystem

Implements the [Kadena Stratum protocol](https://gist.github.com/mightybyte/f1567c2bec0380539c638225fb8c1cf4) (JSON-RPC 2.0 over TCP):

- `Worker/POW/Stratum/Protocol.hs` — message types, JSON codecs
- `Worker/POW/Stratum/Server.hs` — TCP server, per-client state, job broadcasting
- `Worker/POW/Stratum.hs` — glue that turns the "worker" into a **submission endpoint**: the outer `miningLoop` calls `submitWork`, which publishes the work to all connected ASICs and waits for any of them to submit a valid share

`--stratum-difficulty` accepts `block` (use the actual block target) or an integer log2 difficulty ≥ 42. `--stratum-rate` throttles job emission per worker thread so ASICs don't exhaust a job's 280 TH solution space between rebroadcasts.

## External surfaces

- **Outbound:** chainweb-node mining API over HTTP(S). Default `--node localhost:1848`, `--tls` + `--insecure` flags, configurable timeout.
- **Inbound (stratum mode):** TCP on `--stratum-port` (default 1917), binds `--stratum-interface` (default `*`). Plain TCP — no TLS, no auth beyond the stratum-subscribe handshake.
- **Inbound (on-demand mode):** HTTP on `--on-demand-port`, binds `--on-demand-interface`. Single route: `POST /make-blocks` with `{ "0": 5, "1": 3, ... }`.
- **Keys:** `--public-key` + optional `--account`; `--generate-key` prints a new Ed25519 pair to stdout. Private key stays outside the client.

## Workflow quirks

- `main/Main.hs` is ~850 lines — most of them configuration, keygen, and `miningLoop`. Don't be intimidated; the interesting glue is `run` + `withWorker`.
- Test suite does **not** link the executable's modules — it re-compiles `src/` from source (`hs-source-dirs: test, src`). Fresh builds per invocation; slower but insulates tests from exe-only changes.
- `configuration-tools` calls `runWithPkgInfoConfiguration` which is why `--info` / `--long-info` / `--license` / `--version` all exist without explicit wiring.

## Known weak points

- `--stratum-difficulty` boundary conditions (values < 42) are documented as "clamped to at least 42 and at most the block target" — if a downstream chain (StoaChain) lowers block difficulty floor, this clamp may need a second look.
- `Main.hs`'s `miningLoop` uses `retry` with network exceptions; exhausted retries throw. A very long node outage upstream surfaces as a crash rather than graceful backoff.
- Non-PoW modes silently produce rejected blocks on a normal node. That's the README's loudest warning, but it stays a footgun.
