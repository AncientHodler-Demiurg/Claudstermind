# Conventions — ChainwebMiningClient

> Project-specific norms that override or extend the cluster-wide conventions in [`../../meta/shared-conventions.md`](../../meta/shared-conventions.md).

## Overrides

- **Commits / push policy.** Cluster default assumes projects are owned by Mihai and pushed under his PAT. This repo's `origin` is `kadena-io/chainweb-mining-client` — **do not push to `origin/main`**. Any local change that needs to be shared must go to a fork first (under `StoaChain/` or the owner's account). Default assumption for this project is **read-only clone**.
- **Version bumping.** Cluster convention in some projects (notably AncientHoldings) is "bump a dev-suffix on every code change." Does not apply here — version lives in the cabal file and follows upstream Kadena releases. If the owner ever forks, re-evaluate.
- **"Claude owns the worker."** Irrelevant here — there is no dev worker / hot-reload loop for this project. `cabal build` on demand, that's the whole dev cycle.

## Extensions

- **The `Worker` type is sacred.** Any new mining mode must (a) add a constructor to `WorkerConfig` in `main/Main.hs`, (b) add a case in `withWorker`, (c) conform to `type Worker = Nonce -> Target -> ChainId -> Work -> IO Work`. Parallel mining loops that bypass the outer `miningLoop` are out of scope.
- **Non-PoW modes must self-label.** The three non-PoW workers (`simulation`, `constant-delay`, `on-demand`) tell the user they produce invalid blocks unless `DISABLE_POW_VALIDATION=1` is set on the node. A new non-PoW mode must match that pattern — the README warning is load-bearing.
- **Config round-trip.** Every field goes through `--print-config` (YAML) → `--config-file`. New fields must supply both FromJSON and ToJSON plus a default in `defaultConfig`, or the CLI starts emitting unreadable configs.
- **Field / JSON key naming.** `_configFoo` on the Haskell side, `foo` in the emitted YAML/JSON — imposed by `configuration-tools`. Don't invent a third casing.
- **GHC compatibility.** Upstream tests GHC 8.10.7 through 9.4. If a local change needs a newer GHC feature, confirm with the owner first — the cluster's Nix shell defaults to 8.10.7 (`flake.nix`), so anything past that breaks `nix develop`.
- **Don't rewrite `PkgInfo.hs`.** It's autogenerated by the custom-setup. Changes go through `chainweb-mining-client.cabal` / `Setup.hs`, not the generated file.

## Cluster-wide rules that still apply

- Label speculation vs fact — no guessing dressed up as truth.
- Every manual workaround Claude does is a bug on the roadmap. In this repo that mostly means: if a config gymnastics is needed to point the client at StoaChain, file the change as an upstream-proposal note in LEARNINGS rather than patching silently.
- GitHub credentials are stored globally via `credential.helper store`. If a fork is created, it uses the same PAT.
