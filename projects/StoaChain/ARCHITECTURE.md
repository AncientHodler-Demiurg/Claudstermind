# Architecture — StoaChain

> Big-picture design that takes reading several files to grasp. File-by-file navigation is covered in `CLAUDE.md`; this doc is the mental model.

## Stack

- **Language:** Haskell, GHC **9.10.1** + Cabal **3.14.1.1** (hard pins — the custom-setup in `node/chainweb-node.cabal` rejects older cabal 3.10).
- **Protocol:** braided parallel-chain Proof-of-Work (Chainweb), forked from `kadena-io/chainweb-node`.
- **Smart-contract VM:** Pact (both major versions live in-tree — Pact 4 infrastructure retained, Pact 5 is the current execution layer).
- **Storage:** RocksDB (via `rocksdb-haskell-kadena`, pinned in `cabal.project`).
- **Warning flags:** `-Wall -Werror` — strict, no drift allowed.

## Top-level layout

```
stoa-chain/
├── src/Chainweb/              ← library core: cuts, headers, mempool, pact, p2p, consensus, REST
│   ├── Version/Stoa.hs        ← our ChainwebVersion — bootstraps, genesis, gas, forks, verifiers
│   ├── BlockHeader/Genesis/   ← generated Haskell modules (Stoa0Payload, Stoa1to9Payload) — block 0
│   ├── Pact4/ and Pact5/      ← two Pact execution layers, both live
│   └── …
├── node/                       ← thin executable wrapping the library; `main` registers the version
├── cwtools/                    ← CLI utilities (ea = genesis generator, compact, genconf, etc.)
├── libs/chainweb-storage/     ← in-tree RocksDB wrapper
├── pact/                       ← Pact sources consumed by `cwtools/ea` to produce genesis payloads
│   ├── stoa-coin/new-coin.pact ← genesis coin module (FROZEN)
│   ├── stoa-coin/upgrades/    ← read-only snapshots of on-chain upgraded modules
│   └── genesis/stoa/          ← YAML + .pact source feeding ea (FROZEN)
├── test/                       ← 4 test suites (unit, compaction, multi-node, remote)
├── bench/                      ← tasty-bench benchmarks
├── rewards/miner_rewards.csv  ← emission schedule (hash pinned in code)
├── deploy.sh                   ← Ubuntu 22 end-to-end bootstrap
├── run-stoa.sh                 ← canonical production flag set
├── stoa-node.service           ← systemd unit used in production today
├── Dockerfile                  ← multi-stage; `stoa-node` target = hub-ready container
├── docker/entrypoint.sh        ← env-var → CLI-flag translator (authoritative env-var list)
└── HANDOFF.md                  ← current-branch brief for incoming Claude sessions
```

Three packages share one `cabal.project`: `chainweb` (library), `chainweb-node` (executable), `cwtools` (CLIs). Plus the in-tree `libs/chainweb-storage` package.

## Key modules / boundaries

### ChainwebVersion — the network's identity

`src/Chainweb/Version/Stoa.hs` defines `stoa :: ChainwebVersion` plus the `Stoa` pattern synonym. Everything that distinguishes this network lives in that one value: graph (10 chains), block delay, fork heights, genesis payloads, genesis time, bootstrap peers, `_versionMaxBlockGasLimit` (2,000,000), verifier plugins, `_disablePeerValidation` (True — small network, self-signed certs OK).

The value is added to the registry (`src/Chainweb/Version/Registry.hs`) via `registerVersion`, invoked from `node/src/ChainwebNode.hs` after config parsing. The `--chainweb-version stoa` CLI flag is what picks this value at runtime.

### Genesis payloads — the frozen heart

Block 0 is not read from the network — every node computes it locally from compiled-in Haskell modules (`src/Chainweb/BlockHeader/Genesis/Stoa0Payload.hs`, `Stoa1to9Payload.hs`). Those modules are generated by `cwtools/ea` from `pact/stoa-coin/new-coin.pact` and `pact/genesis/stoa/*.yaml`. Any change to the source inputs produces a different block-0 hash; any binary built from changed source cannot sync with the live network.

This is the single most important invariant in the project. On-chain upgrades happen via Pact governance transactions (1-of-7 Stoa Masters keysets, `enforce-one`) that become part of chain history — replayed automatically by any syncing node, no binary change needed.

### Pact integration split across two versions

`src/Chainweb/Pact4/` and `src/Chainweb/Pact5/` coexist. Pact 5 is the current execution layer; Pact 4 code is retained for backward-compat replay of early-chain history. Tests mirror the split (`Chainweb.Test.Pact4.*`, `Chainweb.Test.Pact5.*`). Both Pact libraries are pulled as `source-repository-package` pins in `cabal.project`.

### Miner rewards schedule

`rewards/miner_rewards.csv` holds the emission schedule; its SHA-512 hash is pinned as a constant in code. The two must match. When the schedule changes, both must be updated in the same commit (see commit history around `04-01`).

### P2P + TLS

Bootstrap peers are baked into the `ChainwebVersion` value (`node1.stoachain.com:1789`, `node2.stoachain.com:1789`). Runtime overrides: `--known-peer-info`, `--enable-ignore-bootstrap-nodes`. Peer TLS validation is disabled (`_disablePeerValidation = True`) — intentional for the small network using self-signed certs.

Note: the cluster's AncientHoldings hub has a separate concern about CA-signed certs for P2P operator-box provisioning (see `meta/shared-facts.md` → "Chainweb P2P needs CA-signed TLS"). That applies to how the *hub* helps operators provision node certs; the node itself accepts self-signed peers.

### Container layer (new, `AncientStoa` branch)

Upstream Kadena Dockerfile inherited with a full multi-stage build (builder + test runner + bench + tool stages). We appended a `stoa-node` stage at the end that extends `chainweb-node` (untested image, fast build) with `docker/entrypoint.sh` — a shell wrapper that translates ~40 documented env vars into chainweb-node CLI flags. Because it's the last stage, `docker build .` defaults to it. The `chainweb-node-tested` target (runs tests during build) is still available.

## Data model

Chainweb's storage is RocksDB column families for:
- Cut state (the tip of each of 10 chains + cross-chain adjacency).
- BlockHeader database (one per chain).
- Pact state (execution results, module table, balances — one RocksDB "domain" per chain).
- Mempool (in-memory + persisted).

Important for ops: the Pact state **is not garbage-collected by default** — `FULL_HISTORIC_PACT_STATE=true` is the production setting, baked as the default in `docker/entrypoint.sh`. Turning it off changes replay semantics.

## External surfaces

- **P2P port 1789 (TLS).** Peer-to-peer protocol; self-signed certs accepted.
- **Service API port 1848 (HTTP).** Public RPC — block/tx queries, `send` / `local` endpoints, `/health-check`, mining coordination. In the Docker default, bound to `127.0.0.1` unless `SERVICE_INTERFACE=0.0.0.0` is set.
- **Mining stratum.** Via `chainweb-mining-client` — talks to the service API's mining coordination endpoints when `--enable-mining-coordination` is on.
- **Upstream pins** in `cabal.project` (pact, pact-5, pact-json, rocksdb-haskell-kadena, kadena-ethereum-bridge, wai-middleware-validation, ixset-typed, base64-bytestring-kadena) are specific git commits with SHA256 hashes. Bumping any is a coordinated change.

## Workflow / execution model

Production node lifecycle:

```
chainweb-node starts
  → register 'stoa' version
  → load genesis payloads (Stoa0Payload, Stoa1to9Payload)
  → compute block 0 locally
  → open RocksDB at --database-directory
  → dial bootstrap peers (node1/node2.stoachain.com)
  → begin cut sync
  → once caught up, start mining (if --enable-node-mining) or serve API
```

The native production runtime is `stoa-node.service` systemd unit invoking `run-stoa.sh`. The containerised runtime (in progress on `AncientStoa`) replaces the binary with `docker run stoa-node:latest` and the flag list with env vars.

## Known weak points

- **Dep-graph fragility.** Upstream pins in `cabal.project` (pact, pact-5, crypton, memory, merkle-log, rocksdb-haskell-kadena) form a tight mesh. Small bumps cascade; the `AncientStoa` branch has a live example — the last 4 commits are a back-and-forth pinning crypton 1.0.4 + memory 0.18.0 + merkle-log 0.2.0 against a kda-community freeze file to recover from a GHC-9.10 compatibility wobble.
- **Genesis regeneration is a footgun.** `cwtools/ea` must NEVER be run to "refresh" genesis on a branch that will merge to main. It would silently change block 0. The `check-git-clean.sh` tool in the Dockerfile catches this during CI-style builds.
- **Gas-price ramp not wired.** Functions exist in the live on-chain coin module (per `pact/stoa-coin/upgrades/live-coin-module.pact`) but the protocol minimum-gas-price isn't driven by them yet.
- **Single-node-per-machine assumption in ops scripts.** `deploy.sh` / `run-stoa.sh` / `stoa-node.service` assume one node per host. The container layer doesn't change that — port mappings still collide if you try to run two.
